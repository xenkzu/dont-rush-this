Building an Endless Drag Grid in React, without WebGL

Create an infinite, draggable grid using only the DOM and pointer events — no canvas, no WebGL, no heavy libraries. Just pure math and a clever tiling trick.

WebGL and canvas-based renderers are the first thing most developers reach for when building an infinite drag surface. They're powerful, sure — but they come with real costs: a steep learning curve, large bundle sizes, and complex shader pipelines. For most UI use cases, you simply don't need them.

The trick is to simulate the illusion of infinity with a finite number of DOM elements. The grid you're about to build feels endless in every direction, but under the hood it's tiling a fixed set of cells and translating them as you drag.

The core idea

The technique is called modular offset. Rather than storing the position of every cell in an ever-growing coordinate space, you translate the entire grid container and use the modulo operator to wrap positions back into range when they drift too far.

Think of it like a treadmill: the belt moves, but its length stays constant. Each cell's rendered position is always within the visible viewport, but it appears to come from an infinite surface.

approach
01

Define a tile grid

Render a fixed N×M grid of cells, large enough to cover the viewport plus some overflow buffer on each side.

02

Track a raw offset

On pointer move, accumulate the drag delta into a raw offsetX / offsetY. This value grows freely without bounds.

03

Wrap with modulo

Translate each cell's position using ((col * size + rawOffset) % tileWidth + tileWidth) % tileWidth — the double-mod handles negative values cleanly.

04

Apply lerp for smoothness

Lerp the rendered offset toward the raw offset on every animation frame for a satisfying kinetic feel.

Setting up the hook

First, let's build a useInfiniteGrid hook that tracks the drag state and exposes a smooth transform value via requestAnimationFrame.

TypeScript 
import { useRef, useEffect, useCallback } from 'react'

type Vec2 = { x: number; y: number }

export function useInfiniteGrid() {
  const raw    = useRef<Vec2>({ x: 0, y: 0 })
  const smooth = useRef<Vec2>({ x: 0, y: 0 })
  const dragging = useRef(false)
  const last   = useRef<Vec2>({ x: 0, y: 0 })
  const worldRef = useRef<HTMLDivElement | null>(null)
  const rafRef = useRef<number>(0)

  const lerp = (a: number, b: number, t: number) =>
    a + (b - a) * t

  const tick = useCallback(() => {
    smooth.current.x = lerp(smooth.current.x, raw.current.x, 0.1)
    smooth.current.y = lerp(smooth.current.y, raw.current.y, 0.1)

    if (worldRef.current) {
      worldRef.current.style.transform =
        `translate(${smooth.current.x}px, ${smooth.current.y}px)`
    }

    rafRef.current = requestAnimationFrame(tick)
  }, [])

  useEffect(() => {
    rafRef.current = requestAnimationFrame(tick)
    return () => cancelAnimationFrame(rafRef.current)
  }, [tick])

  // pointer handlers omitted for brevity

  return { worldRef }
}

The wrapping function

Here's the key piece. Each cell's left and top positions are wrapped so they stay within the tile bounds:

TypeScript 
/**
 * Wraps a value within [0, range) — works correctly for negatives.
 * The double-mod pattern handles JavaScript's signed modulo.
 */
const wrap = (value: number, range: number) =>
  ((value % range) + range) % range

// For each cell at grid index (col, row):
const x = wrap(col * CELL_SIZE + smooth.x, COLS * CELL_SIZE)
const y = wrap(row * CELL_SIZE + smooth.y, ROWS * CELL_SIZE)

cell.style.transform = `translate(${x}px, ${y}px)`

Rendering the grid

The component itself is straightforward. We pre-create a flat array of cell elements and position them absolutely inside a world container:

TSX 
const COLS = 5
const ROWS = 5
const CELL = 160 // px
const GAP  = 12  // px

export function InfiniteGrid() {
  const { worldRef } = useInfiniteGrid()

  return (
    <div className="canvas">
      <div className="world" ref={worldRef}>
        {Array.from({ length: COLS * ROWS }, (_, i) => {
          const col = i % COLS
          const row = Math.floor(i / COLS)
          return (
            <Cell
              key={i}
              col={col}
              row={row}
              size={CELL}
              gap={GAP}
            />
          )
        })}
      </div>
    </div>
  )
}

Why not Framer Motion?

You can absolutely integrate Framer Motion for the spring animation — useMotionValue and useSpring work great here. But for this article, I wanted to show that the core mechanism works with zero dependencies: just requestAnimationFrame and a lerp.

The performance is excellent because we never re-render the component tree on drag. All positional updates happen by mutating DOM style directly in the animation loop — no React state, no diffing.

Key insight: Keep the animation loop in a useRef-based RAF loop, not in React state. React reconciliation on every frame would tank performance immediately.

What's next

This is the foundation. From here you can add item selection, multi-select with a rubber band, zoom with a scale transform, or snap-to-grid by rounding the raw offset. The pattern scales well because the rendering cost stays O(COLS × ROWS) regardless of how far the user has dragged.
